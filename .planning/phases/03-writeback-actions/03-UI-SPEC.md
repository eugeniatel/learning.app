Inherited from Phase 1 and Phase 2 UI-SPEC tokens. Extended coral usage documented below.

Source: globals.css :root tokens (session 1, unchanged).

| Role | Token | Hex | Usage |
|------|-------|-----|-------|
| Dominant (60%) | --background | #FAF9F5 | Page background |
| Secondary (30%) | --sidebar | #F5F3ED | Sidebar background |
| Card surface | --card | #FFFFFF | Session cards, question item surfaces |
| Muted surface | --muted / --secondary | #F1EFE9 | Status badges, reflection section bg, artifact chip bg, concept group header bg |
| Muted text | --muted-foreground | #6B675E | Module labels, action button secondary state, status badge text, artifact type labels |
| Border | --border | #E6E2D8 | 0.5px borders on cards, sections, dividers |
| Accent (10%) | --ring / coral | #D97757 | RESERVED (see below) |
| Destructive | --destructive | #B4432E | Not used in this phase |

Coral (#D97757) extended in this phase with two new reservations:

Accent (#D97757) reserved for:
- Save button fill when note editor has unsaved changes (inherited from phase 1)
- Focus ring on textareas (via --ring token, automatic)
- "Add question" button fill state (inherited from phase 1)
- Active nav item indicator for /concepts, /weeks, /review, /questions routes
- "Got it" button fill on /review (inherited from phase 2)
- "Save reflection" button fill when the reflection textarea has unsaved changes (new in phase 3; mirrors dirty-save pattern from phase 1 NoteEditor)
- "Switch" confirmation button — "Yes, switch" (inherited from phase 2)
- in_progress status icon on SessionCard (existing: CircleDot uses text-[var(--ring)])

The read-status artifact toggle states (unread, in_progress, read) use grayscale only: empty circle for unread, half-filled circle for in_progress, filled circle for read. Coral is NOT used here. Rationale: coral signals "commit an action now" (dirty save, confirm gesture). The artifact read-status toggle is a state indicator, not a commitment gesture. Grayscale preserves the "coral = act now" metaphor.

Coral is NOT used for: headings, borders, artifact read-status indicator, question status badges (Open/Parked/Answered), question item surfaces, section labels, module group labels, "Park" button, "Answer" button, "Reopen" button.

**`SessionCard`** (satisfies SESSION-01, SESSION-02)

Current state: server component with static StatusIcon (Circle, CircleDot, Check).

Changes: SessionCard becomes a client component. The StatusIcon element becomes a `<button>` with its own click handler that calls the cycle server action. The article element retains all existing styles unchanged.

Status cycle order: todo -> in_progress -> done -> todo (forward-only cycle)

StatusIcon as button:
- Wraps the existing StatusIcon render in a `<button>` tag
- `aria-label`: "Session status: {status}. Click to cycle." (dynamically constructed from current status)
- `type="button"` to prevent form submission
- No visual wrapper: the button has no additional padding or border; the icon itself is the entire hit target
- Minimum touch target: 20x20px icon satisfies desktop-only requirement (44px touch targets are mobile; this is 1024px+ desktop)
- cursor-pointer on the button element
- The rest of the article (title, estimated minutes, artifact chips) is visually unchanged and non-interactive

Optimistic UI:
- SessionCard tracks status in local useState, initialized from the session prop
- On click: immediately update local state to next status (optimistic flip)
- Fire server action (cycleSessionStatusAction) in the background
- On action failure: revert local state to previous status, show no visible error (single-user local tool; transient failures are recoverable on next click)
- completedAt: set to current ISO timestamp when cycling to done; cleared (undefined) when cycling away from done

Server action: `src/lib/actions/cycle-session-status.ts`
- `cycleSessionStatusAction(weekId: string, sessionId: string, newStatus: "todo" | "in_progress" | "done"): Promise<void>`
- Calls `cycleSessionStatus(weekId, sessionId, newStatus)` in progress.ts
- Calls `revalidatePath("/")` after write

progress.ts gains: `cycleSessionStatus(weekId: string, sessionId: string, newStatus: "todo" | "in_progress" | "done"): Promise<void>`
- Reads progress.json, finds the week by weekId, finds the session by sessionId within that week
- Sets session.status to newStatus
- Sets session.completedAt to ISO timestamp if newStatus is "done"; deletes the field if newStatus is not "done"
- Writes back

**`ArtifactChip`** (satisfies ARTIFACT-01)

Current state: plain `<a>` anchor, icon + title + type label.

Changes: add a small circular read-status toggle button to the left of the title. The anchor remains the primary interactive element (opens the artifact URL in a new tab). The toggle button is a separate `<button>` element inside the anchor — this requires the anchor wrapper to become a `<div>` with manual link handling, or the toggle to sit outside the anchor. Design decision: wrap the chip in a relative `<div>`; the link covers the full chip; the toggle sits at the far left as a sibling `<button>` with `e.stopPropagation()` and `e.preventDefault()` on click to prevent the link-out.

Implementation note for executor: render the chip as a flex row with two interactive zones. The toggle button is the leftmost element, `z-10 relative`. The anchor wraps the icon + title + type label (not the toggle button). The anchor onClick does NOT stop propagation; it opens in new tab normally. The toggle onClick stops propagation to prevent the anchor from firing.

Read-status states (three, grayscale only):

| State | Icon | Color | Description |
|-------|------|-------|-------------|
| unread | Circle | text-muted-foreground/50 | Empty circle, strokeWidth 1, size-3.5 |
| in_progress | CircleDot | text-muted-foreground | Half-filled circle indicator, strokeWidth 1.5, size-3.5 |
| read | CheckCircle2 | text-foreground/70 | Filled checkmark circle, strokeWidth 1.5, size-3.5 |

Cycle order: unread -> in_progress -> read -> unread

Toggle button:
- `aria-label`: "Mark artifact as {next status}" (e.g., "Mark artifact as in progress")
- `type="button"`
- No background, no border, no padding — icon only
- cursor-pointer
- 180ms ease-out color transition on state change

Optimistic UI: ArtifactChip becomes a client component. Tracks readStatus in local useState. Immediate optimistic flip on click, fire-and-forget server action, no visible revert on failure (single-user local tool).

Server action: `src/lib/actions/toggle-artifact-status.ts`
- `toggleArtifactStatusAction(artifactId: string, newStatus: "unread" | "in_progress" | "read"): Promise<void>`
- Reads artifacts.json, finds artifact by id, updates readStatus field, writes back atomically
- Calls `revalidatePath("/concepts/[slug]")` after write — use `revalidatePath("/concepts", "layout")` to cover all concept pages

Existing consumers (WeekView session cards show ArtifactChips, ConceptDetail shows ArtifactChips): both already receive the `artifact` object with its `readStatus`. The toggle makes the status interactive without changing the prop interface. Server components that render ArtifactChip must now render a client component — the executor should ensure ArtifactChip is imported as a client component where needed.

**`ReflectionPrompt`** (satisfies REFLECT-01)

Current state: static section with disabled textarea and a "saving enabled later" note.

Changes: becomes a client component with enabled textarea + "Save reflection" button. Mirrors NoteEditor pattern from phase 1.

Layout:
- Section container: existing `mt-10 rounded-xl bg-secondary/60 p-6` (unchanged)
- Heading: "End-of-week reflection" (14px weight-500, unchanged)
- Sub-copy: existing helper copy (unchanged) — remove the "Saving is enabled in a later session." line
- Textarea: remove `disabled` attribute; add dirty-state tracking; min-h-[128px] resize-y bg-background text-[14px] leading-7 (existing styles, enabling removes disabled)
- Save button: right-aligned below textarea, gap-2 layout
  - Dirty state (has unsaved changes): `bg-[var(--ring)] text-white hover:bg-[var(--ring)]/90` (coral fill, matches NoteEditor dirty pattern)
  - Clean state (saved or no changes yet): variant outline, text-muted-foreground
  - Label: "Save reflection" (dirty or clean)
  - "Saved" label: shown for 2 seconds after successful save, then reverts to "Save reflection" (same as NoteEditor)
- Error state: inline text below save button, text-destructive text-[13px], "Could not save. Try again."

Load behavior: on mount, load existing reflection from `notes/reflections/week-{weekId}.md` (frontmatter stripped, body only). Pre-populate textarea. Server renders the initial value as a prop passed down from the page server component (via `readReflection(weekId)` in the new `src/lib/reflections.ts`).

Write target: `notes/reflections/week-{weekId}.md` with frontmatter `{ weekId, kind: "week-reflection", updatedAt }`.

New lib: `src/lib/reflections.ts`
- `readReflection(weekId: string): Promise<string>` — reads the markdown file, returns body only (empty string if not found)
- `saveReflection(weekId: string, body: string): Promise<void>` — writes to notes/reflections/ subdir, creates dir if needed, gray-matter stringify with frontmatter

Server action: `src/lib/actions/save-reflection.ts`
- `saveReflectionAction(weekId: string, body: string): Promise<void>`
- Calls `saveReflection(weekId, body)` in reflections.ts
- No revalidatePath needed (the textarea stays on the same page, value is already in client state)

## Interaction Contracts

### Session status cycle (SESSION-01, SESSION-02)

- SessionCard becomes a client component (add "use client")
- Props unchanged: `{ session: Session, artifacts: Artifact[], upNext: boolean, weekId: string }` — weekId is added as a new required prop so the action knows which week to update
- WeekView passes weekId to each SessionCard (weekId is already available in WeekView props as week.id)
- Status transitions: todo -> in_progress -> done -> todo
- Click handler: compute nextStatus, update local state immediately, call cycleSessionStatusAction(weekId, sessionId, nextStatus) without await
- On action success: revalidatePath("/") fires server-side; client state already reflects the new status
- On action failure: revert local state to previous status using a try/catch around the awaited action
- "Up next" label: computed server-side and passed as upNext prop; after a status change the upNext indicator will re-sync on the next server revalidation (acceptable for a single-user local tool — no instant re-computation needed client-side)

### Reflection save (REFLECT-01)

- ReflectionPrompt receives `weekId: string` and `initialValue: string` as props
- initialValue is loaded server-side from readReflection(weekId); empty string if no file exists
- textarea tracks dirty state: isDirty = currentValue !== initialValue
- "Save reflection" button enabled when isDirty is true; no save attempted when not dirty
- On save: call saveReflectionAction(weekId, currentValue)
- On success: update initialValue reference to currentValue (set with useRef or re-base), show "Saved" label for 2 seconds, clear isDirty
- On failure: show error text, leave isDirty true so user can retry

### Artifact read-status toggle (ARTIFACT-01)

- ArtifactChip becomes a client component
- readStatus cycles: unread -> in_progress -> read -> unread
- Existing consumers pass the artifact object (which includes readStatus); no prop interface change needed
- Click on toggle: compute nextStatus, update local state, fire toggleArtifactStatusAction(artifact.id, nextStatus) without await
- Revert on failure (catch block)
- The main anchor href behavior is unaffected; it still opens artifact.url in a new tab

### Questions queue (QUESTION-02, QUESTION-03)

- /questions page is a Server Component; loads all questions from getProgress().openQuestions at render time
- Groups questions by conceptId; fetches concept data for each conceptId to get title, slug, and moduleId
- Passes grouped data as props to a QuestionsQueue client component that manages optimistic state for all items
- Park action: updates question status from "open" to "parked" via updateQuestionStatusAction(questionId, "parked"); revalidatePath("/questions")
- Answer action: updates question status from "open" to "answered" via updateQuestionStatusAction(questionId, "answered"); revalidatePath("/questions")
- Reopen action: updates question status from "parked" or "answered" back to "open" via same action; revalidatePath("/questions")
- Server action: `src/lib/actions/update-question-status.ts`
  - `updateQuestionStatusAction(questionId: string, newStatus: "open" | "parked" | "answered"): Promise<void>`
  - Calls `updateQuestionStatus(questionId, newStatus)` in progress.ts
- progress.ts gains: `updateQuestionStatus(questionId: string, newStatus: "open" | "parked" | "answered"): Promise<void>`
  - Reads progress.json, finds question by id, updates status field, writes back

## Transition and Motion

Inherited from Phase 1 and Phase 2 UI-SPECs. No additions.

- All interactive state changes: 180ms ease-out (globals.css a, button rule)
- Status icon state change: 180ms ease-out color/opacity transition
- Save button dirty/clean swap: immediate text/color change, 180ms background transition
- "Saved" label: immediate text change, 2 second timer, revert
- Question item action buttons: 180ms ease-out opacity on state change
- No page transitions, no skeleton loaders, no spinners
- No animations beyond 250ms (project constraint)

## Write Actions

Four new server action files, all following the pattern from src/lib/actions/save-note.ts:

| File | Function | Writes |
|------|----------|--------|
| src/lib/actions/cycle-session-status.ts | cycleSessionStatusAction(weekId, sessionId, newStatus) | progress.json sessions[].status + completedAt |
| src/lib/actions/save-reflection.ts | saveReflectionAction(weekId, body) | notes/reflections/week-{id}.md |
| src/lib/actions/toggle-artifact-status.ts | toggleArtifactStatusAction(artifactId, newStatus) | artifacts.json artifact.readStatus |
| src/lib/actions/update-question-status.ts | updateQuestionStatusAction(questionId, newStatus) | progress.json openQuestions[].status |

All use "use server" directive. Business logic lives in src/lib/progress.ts, src/lib/reflections.ts (new), and a new artifacts write helper. Action files remain thin wrappers.

New lib files:
- `src/lib/reflections.ts` — readReflection(weekId), saveReflection(weekId, body)
- Artifacts write helper: add `writeArtifactStatus(artifactId, newStatus)` to `src/lib/data.ts` or a new `src/lib/artifacts.ts` — executor's choice; follow the pattern in progress.ts

New progress.ts functions:
- `cycleSessionStatus(weekId, sessionId, newStatus)` — finds session in nested week.sessions, updates status + completedAt
- `updateQuestionStatus(questionId, newStatus)` — finds question by id in openQuestions, updates status

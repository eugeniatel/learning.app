Inherited from Phase 1 UI-SPEC. No changes.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px (1) | Icon-to-label gaps, inline chip padding |
| sm | 8px (2) | Compact element spacing, tight gaps |
| md-tight | 12px (3) | Tight gap between row content and confirmation strip, inline metadata clusters (WeekRow left content, WeekSwitchConfirm strip, ReviewCard action row) |
| md | 16px (4) | Default element spacing, card padding |
| lg | 24px (6) | Section padding, between content blocks |

Exceptions: shell-x (40px) and md-tight (12px) are named exceptions to the 8-point scale. md-tight documents the 12px gaps used in WeekRow, WeekSwitchConfirm, and ReviewCard rather than leaving them undocumented. badge-pad (px-1.5 py-0.5) is the shadcn badge micro-padding convention, used on the "Current" badge and module tag. All other spacing uses the 8-point scale.

| Role | Token | Hex | Usage |
|------|-------|-----|-------|
| Dominant (60%) | --background | #FAF9F5 | Page background |
| Secondary (30%) | --sidebar | #F5F3ED | Sidebar background |
| Card surface | --card | #FFFFFF | Week row cards, review concept card |
| Muted surface | --muted / --secondary | #F1EFE9 | Module group headers, inline confirmation bg, review free-text bg |
| Muted text | --muted-foreground | #6B675E | Date ranges, module labels, "last reviewed" text, helper text |
| Border | --border | #E6E2D8 | 0.5px borders on cards, rows, dividers |
| Accent (10%) | --ring / coral | #D97757 | RESERVED (see below) |
| Destructive | --destructive | #B4432E | Not used in this phase |

Coral (#D97757) is extended in this phase with one new reservation:

Accent (#D97757) reserved for:
- "Got it" (yes) button fill on /review (new in phase 2 — primary review CTA)
- Save button fill when note editor has unsaved changes (inherited from phase 1)
- Focus ring on textareas (via --ring token, automatic)
- "Add question" button fill state (inherited from phase 1)
- Active nav item indicator for /weeks and /review routes

Justification for extending coral to the review "yes" button: the review surface is the only page where the primary user action is a commitment gesture ("I know this"). Using coral here gives it the same visual weight as a dirty-save: both are "complete this action now" moments, not navigation. The "Not yet" secondary button stays as variant outline (no coral), preserving the 10% rule.

Coral is NOT used for: "Not yet" button, week row items, module group labels, borders, headings, metadata.

**`WeekRow`** (used in /weeks list)

- Container: full-width row, bg-card rounded-xl border border-border, px-5 py-4
- Layout: flex items-center gap-3 between left content and right action area
- Left content: stacked — week label (12px, muted-foreground) on top, module title (15px weight-500, foreground) below
- Week label format: "Week {number}, {date range}" — uses same formatRange() logic as WeekHeader
- Module title: 15px weight-500 text-foreground
- Session count: 13px text-muted-foreground, format: "{done}/{total} sessions done", right-aligned
- "Current" badge: if this week matches progress.currentWeek, show a small muted badge "Current" (bg-muted text-muted-foreground text-[12px] rounded-sm px-1.5 py-0.5) inline after the week label
- Hover: entire row bg transitions bg-card to bg-muted/40 (180ms ease-out). No shadow.
- The row is NOT a link. Clicking the row triggers the inline confirmation (see Interaction Contracts).

---

**`WeekSwitchConfirm`** (inline confirmation inside WeekRow, satisfies WEEKS-02)

Design decision: use inline confirmation inside the row, NOT a Dialog.

Rationale: switching weeks is a low-stakes operation — it updates progress.json.currentWeek, which Eugenia can switch again at any time. A Dialog is appropriate for irreversible destructive actions (delete, overwrite). Week-switching is reversible and frequent. Inline confirmation matches the "Add question" precedent from phase 1: show the confirmation context in-place, keep the action cheap-feeling. Opening a modal for something so routine would create unnecessary friction.

- State: row has two modes: idle (WeekRow default) and confirming
- Confirming state: row expands below content to show a confirmation strip
- Confirmation strip: bg-muted/60 rounded-b-xl px-5 py-3, flex items-center justify-between
- Confirmation text: "Switch to this week?" text-[13px] text-foreground
- "Yes, switch" button: variant default (bg-primary text-primary-foreground), size sm, text-[13px]
- "Keep current" button: variant ghost, size sm, text-[13px], text-muted-foreground
- On confirm: calls switch-week server action (src/lib/actions/switch-week.ts), updates progress.json.currentWeek, refreshes the page or revalidates the route via router.refresh()
- On cancel: row returns to idle state immediately
- Only one row can be in confirming state at a time (clicking a different row returns previous to idle)

**`ReviewCard`** (used in /review, satisfies REVIEW-01, REVIEW-02)

- Container: single card, bg-card rounded-xl border border-border, px-6 py-6, max-w-prose centered on page
- Header area: concept title (22px weight-500 text-foreground, mb-1) + one-liner (15px weight-400 text-muted-foreground, mb-0)
- "Last reviewed" line: if concept has been reviewed before — "Last reviewed {date}" — text-[12px] text-muted-foreground mt-2. Format: "Last reviewed Apr 14" (month abbrev + day, no year if same year). If never reviewed: "Not yet reviewed" in same style.
- Module tag: 12px bg-muted text-muted-foreground rounded-sm px-1.5 py-0.5, shown below the last-reviewed line, mt-1
- Action row: mt-6, flex gap-3
  - "Got it" button: variant default using coral fill — bg-[var(--ring)] text-white, hover:bg-[var(--ring)]/90, size default, flex-1 max-w-[160px]
  - "Not yet" button: variant outline, size default, flex-1 max-w-[160px]
- Free-text field: mt-4, full-width textarea, min-h-[80px], bg-muted/40 border border-border rounded-lg px-3 py-2 text-[15px] text-foreground, placeholder "Add a note about this concept..." resize-none. Saved with the review entry on submit (either button).
- Transition: card content fades out and new card fades in on submit (150ms ease-out opacity transition, replace content in-place — no page navigation between cards)
- Progress counter: above the card, text-[12px] text-muted-foreground, "{N} concepts in queue" — shown when queue has 2 or more. When only 1 remains, shows "Last one."

---

**`ReviewQueue`** (composite, /review)

- Page heading: "Review" — 22px weight-500, mb-6
- Layout: single column, max-w-3xl, content centered with the ReviewCard taking max-w-prose
- Queue order: least-recently-reviewed first; never-reviewed concepts come last (recency ascending, nulls last). Deterministic sort — no random ordering.
- Renders one ReviewCard at a time
- On submit (either "Got it" or "Not yet"): writes review entry via server action, advances to next card
- Two empty states (see Copywriting Contract below)

## Interaction Contracts

### Week switching (WEEKS-01, WEEKS-02)

- /weeks is a Server Component; initial data fetched server-side via getProgress() + getModules()
- WeeksList is a Client Component to manage confirming-row state
- Clicking any WeekRow (that is not current) sets that row into confirming state; all other rows collapse to idle
- Clicking the current-week row does nothing (no confirmation strip, no action — it is already current)
- Confirming "Yes, switch": calls src/lib/actions/switch-week.ts server action
  - switchWeekAction(weekId: string): void
  - reads progress.json, sets progress.currentWeek to {id, moduleId, number} of the selected week, writes back
  - calls revalidatePath("/") and revalidatePath("/weeks") to reflect the change
- After action resolves: row returns to idle, "Current" badge moves to the newly selected row
- No optimistic UI for week switching — the operation is infrequent and the feedback delay is acceptable
- Pressing Escape while a row is in confirming state collapses it to idle

### Review flow (REVIEW-01, REVIEW-02, REVIEW-03)

- /review is a Server Component; loads all concepts + current reviews from progress.json
- Derives review queue: sort concepts by lastReviewed ascending (null = never reviewed, sort last)
- ReviewQueue is a Client Component that receives the ordered concept list as a prop
- Tracks current index in useState; no server round-trip between cards
- On submit (either button):
  - Calls src/lib/actions/submit-review.ts server action
    - submitReviewAction(conceptId: string, status: "known" | "needs_review", note?: string): void
    - reads progress.json, finds or creates the review entry for conceptId
    - sets lastReviewed to current ISO timestamp
    - sets nextSuggested to 1 day later (ISO timestamp) — simple flat cadence, no spaced-repetition in v1
    - sets status to "known" or "needs_review" based on button pressed
    - if note is non-empty, appends note text to the review entry (note field: string | undefined)
    - writes back to progress.json
  - On action resolve: advance to next card (increment index)
  - Free-text field clears when advancing to next card
- "Got it" maps to status: "known"
- "Not yet" maps to status: "needs_review"
- Keyboard shortcuts: none in phase 2 (deferred; keyboard navigation adds complexity without clear need for a single-user tool)

### Empty state behavior on /review

Two distinct empty states:

1. Queue is empty because no concepts exist (edge case, should not occur in production): show "No concepts to review." with muted body text.
2. Queue is empty because all concepts have been reviewed today (lastReviewed within the last 24 hours for all): show the "all done" empty state (see Copywriting Contract).

The server derives which empty state applies before passing props to the client component.

## Copywriting Contract

| Element | Copy |
|---------|------|
| /weeks nav label | "All weeks" |
| /weeks page heading | "All weeks" |
| "Current" week badge | "Current" |
| Week row confirmation prompt | "Switch to this week?" |
| Week confirm button | "Yes, switch" |
| Week cancel button | "Keep current" |
| /weeks empty state | "No weeks recorded yet." |
| Week switch failure (server action error) | "Could not switch week. Try again." |
| /review nav label | "Review" |
| /review page heading | "Review" |
| Queue counter (plural) | "{N} concepts in queue" |
| Queue counter (singular) | "Last one." |
| "Got it" button | "Got it" |
| "Not yet" button | "Not yet" |
| Free-text placeholder | "Add a note about this concept..." |
| Last reviewed (with date) | "Last reviewed {Mon D}" |
| Last reviewed (never) | "Not yet reviewed" |
| Review submit failure (server action error) | "Could not save review. Try again." |
| Review empty state (all done) | "You've reviewed everything for now. Check back tomorrow." |
| Review empty state (no concepts) | "No concepts to review." |

Rules: sentence case throughout. No em dashes. No exclamation marks. "Got it" and "Not yet" are intentionally short and conversational — matches the low-stakes nature of the action. "You've reviewed everything for now." uses a contraction deliberately; this is the one place where a small moment of warmth is appropriate without crossing into gamification. "Keep current" replaces "Cancel" as the dismiss label on the week-switch confirmation: it names the outcome rather than the action, which is clearer and avoids the generic-label pattern. Error copy mirrors the Phase 1 precedent ("Could not save. Try again.") — problem description + what to do next, no jargon.

## Write Actions

Two new server action files mirror the pattern from src/lib/actions/save-note.ts:

| File | Function | Writes |
|------|----------|--------|
| src/lib/actions/switch-week.ts | switchWeekAction(weekId: string) | progress.json.currentWeek |
| src/lib/actions/submit-review.ts | submitReviewAction(conceptId, status, note?) | progress.json.reviews[] entry |

Both use "use server" directive at the top of the file. Business logic (read/write/validate) lives in src/lib/progress.ts (extending existing readProgress/writeProgress pattern). The action file is thin: validate input, call lib function, revalidate paths.

Progress.ts will gain two new exported functions:
- switchCurrentWeek(weekId: string): Promise<void>
- upsertReview(conceptId: string, status: string, note?: string): Promise<void>
